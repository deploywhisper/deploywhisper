# DeployWhisper Support

DeployWhisper support is community-oriented and public by default. The project is designed for self-hosted operation, so support requests should include enough local context for maintainers and contributors to reproduce the issue without exposing secrets.

## Where to Ask

- Use GitHub issues for reproducible bugs, documentation gaps, and feature requests.
- Use GitHub discussions, if enabled, for questions, usage patterns, and design discussion.
- Use private vulnerability reporting for security-sensitive issues. See `SECURITY.md`.

## What to Include

For bugs, include:

- DeployWhisper version or commit.
- Operating system and Python version.
- Deployment mode, such as local SQLite, Docker Compose, or PostgreSQL.
- Input toolchain, such as Terraform, Kubernetes, Ansible, Jenkins, or CloudFormation.
- Expected behavior and actual behavior.
- Minimal synthetic sample artifacts or logs with secrets removed.

For feature requests, include:

- The problem you are trying to solve.
- The workflow or toolchain affected.
- Why existing DeployWhisper output is insufficient.
- Whether the request affects advisory output, evidence, context, integrations, docs, or governance.

## Security and Privacy

Do not post secrets, production infrastructure state, real incident records, customer names, private hostnames, or provider credentials in public issues. Use synthetic examples whenever possible.

## Support Boundaries

Maintainers and contributors can help with public project usage, bugs, docs, and contribution review. The project does not guarantee response times, operational on-call coverage, private implementation consulting, or roadmap priority through support requests.

## Roadmap Influence

Roadmap changes should be discussed in public issues, pull requests, benchmarks, RFCs once available, or planning artifacts. Public evidence and community benefit should drive prioritization.
